## Bugs Found

### P1: BUG 2 — Missing Auto-Start Season for Non-Replay Drafts (HIGH)

**Root Cause**: Draft server auto-starts season only for `isReplay===true` leagues. Non-replay leagues are left in PRE_DRAFT after draft completes, requiring manual commissioner action.

**Files**: `lib/draft/server.ts:336`

**Fix**: Remove the `if (league?.isReplay)` guard; `startSeason()` is idempotent and safe to call for all leagues.

**Why**: Asymmetry confuses commissioners. Replay leagues auto-transition; live leagues require manual intervention. Both should auto-transition.

**Effort**: 2 min code change + 5 min test

---

### P1: BUG 3 — Silent Playoff Initialization Failure (MEDIUM)

**Root Cause**: `advanceSeason()` auto-calls `startPlayoffs()` but swallows errors with `.catch()`. If playoff bracket generation fails, the league gets stuck (status="COMPLETE", playoffStatus="NOT_STARTED") with no error signal to the commissioner.

**Files**: `lib/season/index.ts:133-135`, `app/api/leagues/[leagueId]/season/route.ts`

**Fix**: Return `{ scoredWeeks, playoffError?: string }` from `advanceSeason()`. API route checks and returns 500 if playoffError is set.

**Why**: Commissioners need visibility into playoff failures. Current logging is invisible and blocks renewal indefinitely.

**Effort**: 30 min (return type change, API consumer updates)

---

### P2: BUG 4 — No Mode Guard for Playoff Seeding (MEDIUM)

**Root Cause**: `startPlayoffs()` always uses `computeVpStandings()` to seed teams, regardless of league's `scoringMode`. A VTF league entering playoffs will seed by null/zero VP.

**Files**: `lib/services/playoff-service.ts:161`

**Fix**: Add guard before seeding:
```
if (league.scoringMode !== "VP") {
  throw new Error("Playoffs require VP scoring mode.");
}
```

**Why**: Prevents undefined behavior. Playoffs are designed for VP mode; VTF is a different format.

**Effort**: 5 min code + 10 min test

---

### P2: BUG 6 — Non-Transactional Waiver Undo (MEDIUM)

**Root Cause**: `undoWaiverTransaction()` doesn't wrap the roster-entry and event-delete in a transaction. If event delete fails, the entry is deleted but the event remains (orphaned).

**Files**: `app/api/leagues/[leagueId]/commissioner/undo-transaction/route.ts:75-115`

**Fix**: Wrap in `prisma.$transaction([...])`:
```
await prisma.$transaction([
  prisma.rosterEntry.delete(...),  // or create
  leagueEventModel.delete(...)
]);
```

**Why**: Ensures atomicity. Current approach can leave roster mangled if DB error occurs mid-operation.

**Effort**: 10 min

---

### P3: BUG 5 — Draft Persistence Race Condition (MEDIUM, Deferred)

**Root Cause**: Draft state is committed to memory before persistence. If `persistPick()` throws, the in-memory state won't roll back; on restart, the DB state wins. Client might see pick accepted but then reverted.

**Files**: `lib/draft/server.ts:140-154`

**Why**: Not critical because (A) errors are rare, (B) server rebuild from DB is idempotent, (C) engine uses draftedPlayerIds as source of truth. Acceptable for v1.

**Effort**: 60+ min (requires design decision: pending state vs implicit recovery)

**Post-Launch**: Consider monitoring pick rejection rates; if users report reversions, prioritize this.
